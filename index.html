🏗️ Architecture Modulaire

Le système est organisé en couches indépendantes permettant de changer de fournisseur d'API en modifiant une seule variable.

  • ApiProvider — Interface abstraite : chaque fournisseur implémente fetchMatches() et fetchStandings()
  • SyncEngine — Orchestre les appels, gère le cache, les erreurs et les retry
  • DataStore — État centralisé (scores, statuts, classements)
  • UI Layer — S'abonne aux changements du DataStore et se met à jour sans rechargement
  • CacheManager — TTL configurable, évite les appels redondants

🛡️ Gestion des Erreurs & Cache
  • Retry automatique : 3 tentatives avec backoff exponentiel (1s, 2s, 4s)
  • Timeout : 8 secondes par requête, abandon et log de l'erreur
  • Cache TTL : 4 min pour les matchs en cours, 30 min pour les matchs terminés
  • Rate limiting : compteur journalier, alerte à 80% de la limite
  • Fallback : si l'API échoue, les données locales (localStorage) sont conservées
  • Mode dégradé : si l'API est indisponible, la sync s'arrête et notifie l'utilisateur
⚡ Comment ajouter un nouveau fournisseur API

1. Dans API_PROVIDERS, ajouter un objet avec name, fetchMatches(key, tournamentId)
2. La fonction doit retourner un tableau de Match normalisés
3. Sélectionner le fournisseur dans le menu déroulant ⚙️ API
4. Aucune autre modification nécessaire — le SyncEngine s'adapte automatiquement
